Raj's Multi-Agent Workflow
--------------------------

### Phase 1: Planning (Human-Led)

1. **Voice dictation** via Granola during meetings or thinking sessions
2. Transcript processed through **Gemini or Claude** into a structured implementation plan
3. Plan reviewed and refined by Raj before any agents launch

### Phase 2: Agent Architecture (Human-Led)

1. **Manager agent** spun up with a crafted persona — explicitly told not to write code ("you'll be fired if you write any code yourself")
2. Manager decomposes problems, delegates to **specialist agents** (typically 2-5) scoped to specific subsystems
3. A **scribe agent** added for non-blocking documentation (50K+ words of accumulated system knowledge)

### Phase 3: Execution (Agent-Led, Human-Supervised)

1. Specialists work in **separate repo contexts** to avoid conflicts
2. Agents **not allowed to commit** — Raj controls git operations
3. Manager facilitates **knowledge swapping** between specialists: "Expert 1 and Expert 2 look at both codebases, swap knowledge, detail inconsistencies"
4. Raj reviews **plans and approaches**, not line-by-line code

### Phase 4: Integration (Human-Led)

1. Changes submitted as **GitHub PRs**
2. **code-review skill** and **address-code-review skill** run in sequence before pushing
3. Raj reviews PRs via GitHub, leaves comments that agents address

Themes
------

**1. Agent isolation is non-negotiable.** Raj learned through painful experience that agents sharing a working directory causes conflicts. His manual discipline — "I don't have them running in the same repo if possible" and "I don't have them running commit commands" — is exactly what AgenC provides structurally through mission-scoped repo clones. This validates AgenC's core architectural bet.

**2. Hierarchy and scoping are essential for multi-agent work.** "You do need hierarchy... the main value is to say only think about this subsection of the problem." Raj's manager-agent pattern mirrors organizational design — a team lead delegating to specialists.

**3. Knowledge persistence is the next frontier.** Raj has a save-learnings skill writing to Obsidian and is considering session-termination hooks or weekly cronjobs. Ephemeral sessions create real knowledge loss. The tooling hasn't caught up.

**4. Permission prompts are the silent killer of agent productivity.** Raj's most visceral pain point. Agent chunks run 5-10 minutes if unblocked, but permissions stall them indefinitely. Any tool claiming to support unattended agents must solve this.

**5. "Build with, not hack into."** Raj explicitly values AgenC's Unix-philosophy approach — wrapping Claude Code rather than replacing it. This means AgenC's value proposition is resilient to Claude Code updates, directly addressing Raj's concern about staying current.

**6. Planning before coding.** Voice dictation → transcript → refined plan → agent execution. Raj reviewed plans, not code, and shipped production Terraform he'd never written. This plan-first approach is the core of high-leverage AI-assisted development.

**7. Cost efficiency through better architecture.** "My overall Claude costs went down when I started using Agent Teams." More agents cost less because they're scoped and finish faster. Poorly scoped agents thrash and waste tokens.

Agreement and Disagreement
--------------------------

### Aligned On

- Workspace isolation per agent is mandatory, not optional
- Hierarchy in agent organization produces better results than flat peer arrangements
- Plan review over code review — review architecture, not lines
- Unix philosophy — compose focused tools rather than build monoliths
- Permission management is critical infrastructure, not just UX
- The world is moving toward CEO-of-agents workflows

### Diverged On

- **tmux as session manager**: Kevin built AgenC on tmux. Raj prefers Ghostty's GPU-rendered snappiness. "After using tmux, I actually don't like it enough." Not a dealbreaker but a real tension.
- **Agent persistence model**: Raj wants "continually persistent" agents. AgenC's missions are ephemeral by design. Raj's "team member" mental model clashes with AgenC's "contractor" model.
- **Rebuild vs. leverage**: Raj has low tolerance for tools requiring significant setup: "I don't want to rebuild infra... I'd rather leverage the stuff from others." AgenC needs to be turnkey.

Strategic Implications (Creative Director Analysis)
----------------------------------------------------

### ICP Validation

Raj is the strongest ICP match yet. Perfect fit: terminal-native power user, solo operator within team, "Inputs Not Outputs" practitioner, values Unix philosophy. His 36-hour Terraform PR validates the ICP is "people who can do this faster and more reliably with AgenC" — not people who can't do it at all.

**Challenge:** Raj is almost too sophisticated. He's built half of AgenC's value proposition himself. The risk is building exclusively for users who could build their own solution.

### Competitive Position

- **vs. Claude Agent Teams:** AgenC is not replacing Agent Teams but providing the infrastructure layer around it — mission isolation, persistent config, repo library, session lifecycle. This is complementary, not competitive.
- **vs. Gastown:** Raj's friend abandoned custom tmux tooling for Gastown because "software teams are organized." Gastown offers opinionated structure. AgenC should offer structure while preserving flexibility.
- **vs. custom tooling:** The real competition is the user's own bespoke setup. AgenC must convince users that a managed system is worth the switching cost.

| Signal | Strength | Existing Bead |
|--------|----------|---------------|
| Session persistence / agent continuity | Very Strong (3/4 conversations + Raj) | `agenc-1tz0` (partial — capture only, not persistence) |

### Risk Signals

1. **Platform dependency:** Raj's "how quickly can the tool capitalize on new features?" is a purchase objection. If Claude Code ships native features that overlap, AgenC must demonstrate it absorbs them quickly.
2. **Power user trap:** Every interviewee is sophisticated. Feature prioritization may be biased toward advanced needs while the largest market needs simpler things.
3. **"I'll try it" is not adoption.** The 36-hour PR was done without AgenC. The follow-up question is: what would have gone differently with AgenC?
