Mission Reload `--prompt` Flag — Design
========================================

Status: approved (Phase 1 + edge cases + Phase 2)
Author: feat/mission-reload-prompt

Summary
-------

Add a `--prompt` flag to `agenc mission reload <id>` so that a reload can carry a follow-up message that Claude executes after resuming. The headline use case: a mission can self-reload its own Claude *and* feed in an instruction to run on restart.

This is wire-up work — the underlying mechanics already exist:

- `buildWrapperResumeCmd(missionID, prompt)` (internal/server/missions.go:1112) already accepts a prompt and emits `agenc mission resume --run-wrapper <id> --prompt '<escaped>'` with single-quote escaping.
- `runWrapperDirect` (cmd/mission_resume.go:56) already passes `initialPrompt` into `wrapper.NewWrapper`.
- The wrapper passes the prompt as a positional arg to `claude -c` (internal/wrapper/wrapper.go:396), which the user has verified works.

The reload path currently hardcodes `""` as the prompt at three call sites; this change threads a real value through one of them and does light surrounding cleanup.

Scope
-----

In scope:

- New `--prompt` flag on `agenc mission reload`.
- Single-mission only — drops the existing multi-mission/fzf-multi-select path, which is unused (verified: every callsite in the codebase passes a single `$AGENC_CALLING_MISSION_UUID`).
- Per-mission reload guard so two concurrent reloads on the same mission cannot interleave.

Out of scope:

- Prompt support on the headless/no-pane recovery branch. That branch is stale-state recovery and is rare; with `--prompt` set we return a 400 directing the user to a fresh attach.
- Multi-mission reload-with-prompt.
- Queueing or waiting on the per-mission reload guard — it's reject-on-conflict (409).

Phase 1: Architecture and Components
------------------------------------

### Data flow

```
agenc mission reload <id> --prompt "X"
  -> cmd/mission_reload.go: runMissionReload
client.ReloadMission(missionID, prompt)
  -> POST /missions/{id}/reload  body: {"prompt": "X"}
handleReloadMission
  -> resolveMissionID, GetMission, archived check, agent/ check
  -> tryAcquireReloadLock(resolvedID)  -> 409 if busy
  -> defer release()
  -> if mission.TmuxPane present:
       reloadMissionInTmux(missionRecord, paneID, prompt)
         -> verify pane, set remain-on-exit on, defer reset
         -> stopWrapper(missionID)
         -> buildWrapperResumeCmd(missionID, prompt)
         -> tmux respawn-pane -k -t %pane '<resumeCmd>'
    else:
       if prompt != "":
           return 400 "--prompt requires a mission with a live tmux pane"
       stopWrapper(missionID)  // existing stale-state recovery
  -> 200 {"status": "reloaded"}
```

The pool window's pane then runs `agenc mission resume --run-wrapper <id> --prompt 'X'`, which constructs a wrapper with `initialPrompt = "X"`, and the wrapper feeds the prompt to `claude -c`.

### Per-mission reload guard

`Server.reloadsInProgress sync.Map` stores `missionID -> struct{}` for currently-reloading missions. Acquisition uses `LoadOrStore` for atomic test-and-set:

```go
func (s *Server) tryAcquireReloadLock(missionID string) (release func(), ok bool) {
    if _, loaded := s.reloadsInProgress.LoadOrStore(missionID, struct{}{}); loaded {
        return nil, false
    }
    return func() { s.reloadsInProgress.Delete(missionID) }, true
}
```

Properties:

- Memory footprint is O(active reloads), not O(ever-reloaded missions); the entry is deleted on release.
- `defer release()` covers panic safety.
- Server crash drops the map cleanly — no stale-lock recovery needed.
- Different missions can reload concurrently; only same-mission concurrent reloads collide.
- 409 Conflict on collision; the caller decides whether to retry.

Edge Cases (discovered)
-----------------------

The cases below were enumerated against the Phase 1 architecture before Phase 2 was finalized.

### Inputs

- **Empty prompt (`--prompt ""`)** — treat identically to flag absent. `buildWrapperResumeCmd` already skips the `--prompt` append when prompt is empty.
- **Single quotes / shell metachars / `$(...)` / backticks / `;` / `&&`** — all neutralized by the existing single-quote escaping in `buildWrapperResumeCmd` (line 1119: `'` → `'\''`). Tests assert this round-trips literally.
- **Newlines in prompt** — passed through unchanged. Claude itself handles multi-line input.
- **Very long prompt (>ARG_MAX, ~256KB on macOS)** — surfaces an OS exec error from `respawn-pane`. Not pre-validated; the realistic use case won't approach this.
- **`--prompt` without value / with multiple positional IDs** — handled by Cobra's flag parser and `MaximumNArgs(1)`.

### State and transitions

- **Mission archived** — existing 400 stands.
- **Mission has stale pane (DB pane gone in tmux)** — existing 500 from the pane-exists check; prompt silently dropped. Acceptable (rare; user retries).
- **Mission has no pane in DB** — with `--prompt` set, return 400 with a clear message; without `--prompt`, keep current `stopWrapper` recovery behavior.
- **Reload mid-Claude-turn** — `stopWrapper` kills the wrapper before respawn; the prompt arrives on next `claude -c` startup, no race.
- **Mission self-reloads itself** — works the same as prompt-less self-reload today: `respawn-pane -k` kills the calling wrapper before the new resume command starts.

### Concurrency

- **Two reloads on the same mission** — defended by the new per-mission guard (409 Conflict).
- **Two reloads on different missions** — proceed in parallel; guard is per-mission, not global.

### External dependencies (tmux)

- **`respawn-pane` fails** — existing 500; defer cleanup releases the reload guard and resets `remain-on-exit`.
- **`set-option remain-on-exit` fails** — existing 500.
- **Pool window destroyed mid-reload** — caught by the existing pane-exists check before any state mutation.

### Adversarial

- **Prompt content as attack surface** — prompt is consumed by Claude, not by a shell. The shell-side hop is fully escaped. Same exposure as `mission new --prompt`; no new attack surface introduced.

Phase 2: Error Handling and Tests
---------------------------------

### Error matrix

| Failure | HTTP | Message |
|---------|------|---------|
| Mission ID doesn't resolve | 404 | "mission not found: X" (existing) |
| Mission archived | 400 | "cannot reload archived mission" (existing) |
| Old-format mission (no agent/) | 400 | existing message |
| Reload already in progress for this mission | **409** | "reload already in progress for mission X" |
| `--prompt` set on mission with no live pane | **400** | "--prompt requires a mission with a live tmux pane; mission X has none" |
| Pane disappeared mid-reload | 500 | existing "tmux pane X no longer exists"; prompt silently lost on this path |
| `tmux respawn-pane` fails | 500 | existing wrap; defer releases the lock and resets `remain-on-exit` |
| `stopWrapper` fails | 500 | existing wrap; defer releases the lock |
| More than 1 positional arg to CLI | exit 1 | Cobra default from `MaximumNArgs(1)` |

### Tests

Server unit tests (`internal/server/missions_test.go`):

1. `TestBuildWrapperResumeCmd_ThreadsPrompt` — non-empty prompt produces `--prompt '...'` suffix; empty prompt produces no suffix.
2. `TestBuildWrapperResumeCmd_EscapesSingleQuotes` — `don't 'do' it` round-trips literally inside the single-quoted shell argument.
3. `TestBuildWrapperResumeCmd_PreservesShellMetachars` — `$(rm -rf /); echo hi` and similar payloads round-trip literally.
4. `TestTryAcquireReloadLock_Concurrent` — two goroutines acquire the same ID; exactly one gets `ok=true`. After its release, a third acquire succeeds. Acquires for *different* IDs proceed concurrently.
5. `TestHandleReloadMission_PromptOnPanelessMission_Returns400` — mocked mission with `TmuxPane=nil`, body `{"prompt":"x"}`, expect 400.
6. `TestHandleReloadMission_NoPrompt_PanelessMission_StopsWrapper` — same fixture, empty body, expect 200 and `stopWrapper` invoked.
7. `TestHandleReloadMission_ConcurrentSameMission_Returns409` — second concurrent call gets 409.

CLI test (`cmd/mission_reload_test.go`):

8. Adjust existing assertions for single-mission shape. Verify `--prompt "x"` reaches the client `ReloadMission` call.

E2E (`scripts/e2e-test.sh`):

9. `agenc-test mission reload --help` exits 0 and contains `--prompt`.
10. `agenc-test mission reload <bad-id> --prompt "x"` exits nonzero with a recognizable mission-not-found message.

### Manual verification

The end-to-end "reload + prompt arrives in claude" path requires a live tmux session:

- Start a mission, send a few prompts to establish a session, then run `agenc mission reload <id> --prompt "what was my last message?"`. Confirm the pane respawns, claude resumes, and the prompt is automatically submitted.
- Repeat from inside the mission itself (mission reloads its own claude).

Per `CLAUDE.md` tmux-integration rule: this manual step is required; `make build` and `make e2e` passing are necessary but not sufficient.

Implementation Notes
--------------------

- The `_ = ReloadMissionRequest{}` empty struct in missions.go gains a `Prompt` field. JSON-omitempty so old wrappers/clients that send no body or `{}` continue to work (server defaults prompt to `""`).
- The CLI client's `ReloadMission(id)` becomes `ReloadMission(id, prompt string)`. Update the single internal callsite (cmd/mission_reload.go).
- No architecture-doc update required: no package boundaries change, no new process, no schema change. The doc-generator regenerates `docs/cli/agenc_mission_reload.md` automatically.
